# 条件付きアクセス ポリシー更新時に「Update policy」の監査ログが分割されて記録される挙動について

## はじめに

こんにちは、Azure Identity サポート チームの〇〇(他の記事同様名前を入れたい) です。Microsoft Entra ID の監査ログでは、1 回の操作で記録されるデータ量が大きい場合に、監査ログが自動的に複数のレコードに分割されて記録されることがあります。
特に条件付きアクセス ポリシーにおいて多数のターゲット リソースを指定している環境では、ポリシーを 1 回更新しただけであるにもかかわらず、「Update policy」の監査ログが複数件連続して記録され、変更内容の把握が困難になる場合があります。

本記事では、まず監査ログが分割される一般的な仕組みを説明し、次に条件付きアクセス ポリシーでこの事象が発生する具体例と対処方法について解説します。

## 監査ログの分割記録とは

Microsoft Entra ID の監査ログには、1 件のログ レコードに記録可能なデータ サイズの上限がある場合があります。通常の操作では変更内容は 1 件のレコードに収まりますが、記録対象のデータ量がこの上限を超過する場合、データが欠落しないよう、システム側で監査ログが自動的に複数件に分割されて記録されます。

この挙動は想定された動作であり、監査ログの内容そのものが失われているわけではありません。

### 通常のログと分割されたログの違い

通常、変更内容が 1 件のレコードに収まる場合には、JSON の監査ログ レコードの `targetResources` 内の `modifiedProperties` に変更前・変更後の値が直接表示されます。

一方、データ量が大きくなり分割が発生した場合、変更内容は `modifiedProperties` には記録されず、代わりに `additionalDetails` 側に分割されたデータとして記録されます。このため、Microsoft Entra 管理センター上では「変更されたプロパティ」タブが空で表示されますが、これは変更が行われていないことを示すものではありません。

### 分割されたログの構造

分割された監査ログでは、Microsoft Graph API の監査ログにおける `additionalDetails` セクションに、以下のような管理用のキーが含まれます。

| キー | 意味 | 役割 |
|------|------|------|
| id | 対象リソースの識別子 | 分割された複数のログが同一の操作対象に属することを示します |
| seq | シーケンス番号 | 分割されたデータの何番目の断片かを示します (例：1, 2, 3…) |
| b | データ本体 | 分割された JSON データの一部 (断片) です |
| c | 総数 | データが何件に分割されているかを示します |

同一の `id` を持つログを `seq` の順番に並べ、それぞれの `b` の値を順に結合することで、本来 1 回の操作として記録されるべき変更内容を論理的に復元することが可能です。

## 条件付きアクセス ポリシーでの発生例

### 発生する状況

条件付きアクセス ポリシーにおいてターゲット リソース (アプリケーション) を多数指定している場合、ポリシー更新時に記録されるデータ量 (アプリケーション GUID などを含む JSON データ) が非常に大きくなります。その結果、前述の分割記録が発生し、1 回の更新操作に対して「Update policy」の監査ログが複数件生成されます。

### 弊社検証環境での確認結果

参考までに弊社検証環境において、条件付きアクセスの対象ターゲット リソースを約 480 個指定したうえでターゲット リソースの更新を行ったところ、1 回の更新操作に対して「Update policy」の監査ログが 5 件生成される挙動を確認しました。

これらのログは、同一の日時、同一のターゲット、同一の開始者 (アクター) で連続して記録されており、見た目上は「複数回更新が行われた」ように見える点が特徴です。

![](Pasted image 20260318120234.png)

しかしながら、これらのログを 1 つずつ確認すると、複数回の更新が行われたのではなく 1 回の更新が複数の監査ログ レコードに分割されて記録されていることに気づきます。

### API で取得したレコードの例

以下は、分割されたログのうち 1 レコードを Microsoft Graph API で取得した例です。`modifiedProperties` が空であり、変更データが `additionalDetails` 内の `b` キーに断片として格納されていることがわかります。

> [!NOTE]
> 以下の例では説明のために JSON 内にコメントを付与していますが、実際の API レスポンスにコメントは含まれません。

```json
{
  "id": "Directory_xxx_0001",
  "category": "Policy",
  "activityDisplayName": "Update policy",
  "correlationId": "00000000-0000-1234-5678-0000000090ab",
  "activityDateTime": "2026-03-06T06:02:10Z",
  "result": "success",

  "targetResources": [
    {
      "id": "<対象ポリシーの GUID>",
      "displayName": "ExampleConditionalAccessPolicy",
      "type": "Policy",
      "modifiedProperties": []
      // データが大きいため modifiedProperties は空になる
    }
  ],

  "additionalDetails": [
    {
      "key": "id",
      "value": "policy-id-123"
      // 分割された複数のログが、同一の操作に属することを示します
    },
    {
      "key": "seq",
      "value": "1"
      // 分割ログの順序番号 (同一 "id" 内での位置)
    },
    {
      "key": "b",
      "value": "{ \"Users\": { \"Include\": [ { \"Groups\": [ \"group-id-1\" ] } ],"
      // 実際のポリシー設定 JSON の断片 (文字列) 
      // seq 順にすべて結合して初めて意味を持つ
    },
    {
      "key": "c",
      "value": "5"
      // 重要
      // この操作 (correlationId) で生成された分割ログの総数
      // seq=1 ～ seq=5 が存在することを示す
    }
  ]
}
```

## 分割された監査ログの復元方法

### 手動で復元する方法

分割された監査ログの内容を確認する一例として、テキスト エディターを用いてデータを結合する方法をご紹介します。

1. [Microsoft Entra 管理センター] > [監視と正常性] > [監査ログ] を表示します
2. 同一の日時、同一のターゲット、同一の開始者 (アクター) で連続して記録されている複数の「Update policy」ログを確認します
3. 各ログの `additionalDetails` 内にある `b` の値を、`seq` の順にコピーします
4. 空白や改行を入れずに連結し、1 つの JSON 文字列として復元します
5. 復元したデータを確認し、どのような変更が行われたかを確認します

この方法により、分割されて記録された変更内容を確認することが可能です。

### スクリプトで自動的に復元する方法

上述した手動での結合作業は、分割数が多い場合には煩雑になるため、調査を補助する目的でスクリプトを用いた自動化についてもご案内いたします。

分割された監査ログを Microsoft Graph API 経由で自動的に取得・結合し、変更前後の差分を確認する PowerShell スクリプトを以下の GitHub リポジトリにて公開しています。事前準備や実行手順の詳細についてもリポジトリの README に記載しておりますので、併せてご参照ください。

## まとめ

- Microsoft Entra ID の監査ログは、記録されるデータ量が大きい場合に自動的に複数レコードに分割されます。この挙動は想定された動作です
- 条件付きアクセス ポリシーにおいて多数のターゲット リソースを指定している環境では、この分割が発生しやすくなります
- 分割されたログでは、`additionalDetails` 内の `id`、`seq`、`b`、`c` の各キーを基に内容を復元して確認する必要があります
- 「変更されたプロパティ」タブが空であっても変更が行われていないとは限りませんのでご注意ください
